## Bucket 04: rl/ (selector, eligibility, mask, observation, reward)

Scope: all 15 files under `src/agentshore/rl/`. Reviewed in full. Focus areas:
action-masking / eligibility spaghetti, the 246-feature observation vector's
magic-index assembly, reward shaping, and version/dimension coupling across
files.

Summary of conviction: the RL engine is well-factored in places (the
EligibilityAuthority dispatch table, the typed MaskReason) but carries two large
duplicated rule-set surfaces (eligibility/mask split), a fully dead 116-line
parallel mask implementation, a hand-assembled 246-slot observation vector held
together by ~60 magic `_S_*` index constants and three independent block-size
arithmetic chains, and a 19-term reward sum that is triplicated (constant ->
breakdown field -> raw sum -> log call). These are the high-leverage targets.

---

## Critical

### C1. Dead parallel mask implementation — 7 stage functions + 2 helpers, ~116 lines of unreachable production code
**Location:** `mask.py:494-609` (`_stage_preconditions`, `_stage_agent_eligibility`, `_stage_wedged_end_agent`, `_stage_candidate_required`, `_stage_instantiate_config`, `_stage_end_session`, `_stage_take_break`, `_stage_reserved_slots` free fn, `_stage_drain_mode` free fn).
**Problem:** The mask pipeline was reimplemented as `EligibilityAuthority` + `ActionMaskBuilder` (mask.py:298-486). The seven module-level `_stage_*` free functions are the *old* pipeline and are now referenced **only by tests and docstrings** — never by `compute_action_mask`, `compute_mask_reasons`, or any production caller (verified by grep across `src/`). They duplicate, with subtly different logic, the exact gates the authority now owns:
- `_stage_instantiate_config` (mask.py:539) rebuilds `build_candidate_plan(state)` from scratch (ignoring the passed plan) and re-implements the no-active-agents/terminal logic that `EligibilityAuthority._instantiate_config_reason` (eligibility.py:549) already owns.
- `_stage_take_break` (mask.py:580) duplicates `EligibilityAuthority._has_break_trigger` (eligibility.py:576) verbatim.
- `_stage_candidate_required` / `_stage_end_session` / `_stage_wedged_end_agent` duplicate authority steps 4/6 and the wedged re-enable.

Two implementations of the same gates that can silently drift is exactly the failure mode the "single source of truth" docstring (eligibility.py:1) was written to prevent — and the dead copy already *has* drifted (it re-derives the candidate plan instead of taking the shared one).
**Code-judo remedy:** Delete all seven `_stage_*` free functions plus the duplicate `_stage_reserved_slots`/`_stage_drain_mode` free fns (mask.py:494-609) and retarget the ~6 tests in `test_rl_mask.py` (1680-1766) at `EligibilityAuthority`/`ActionMaskBuilder`. Removes ~116 lines and eliminates the second mask code path entirely. Net: one mask implementation, not two.

## High

### H1. Observation version marker computed with a wrong-and-duplicated divisor — confirmed correctness bug + magic coupling
**Location:** `observation.py:576` (`obs[_S_OBS_VERSION] = OBSERVATION_VERSION / 13.0`) vs docstring `observation.py:34` (`version marker = OBSERVATION_VERSION / 12.0`).
**Problem:** The slot-245 version marker divides `OBSERVATION_VERSION` (=13) by a hard-coded literal that must equal the current version to yield 1.0. The docstring says `/12.0`, the code says `/13.0` — they already disagree, proving this literal is forgotten on every bump. Worse, the divisor being a free literal means a v14 bump that misses line 576 silently feeds the network a marker of 14/13 = 1.077 (an out-of-[0,1] feature the policy never saw in training) with no error. This is a magic number coupled to a version constant living three lines away.
**Code-judo remedy:** Replace the literal with `obs[_S_OBS_VERSION] = 1.0` (the marker only needs to be a stable per-version constant; the current scheme always intends 1.0) — or `OBSERVATION_VERSION / float(OBSERVATION_VERSION)`. Either deletes the bug class permanently and removes the doc/code divergence. Fix the docstring at line 34 regardless.

### H2. The 246-slot observation vector is hand-assembled from ~60 magic `_S_*` indices with three independent block-arithmetic chains — fragile, version-coupled, no integrity check beyond a single assert
**Location:** `observation.py:90-224` (the `_S_*` constant wall + three block-size derivations) and the 18 hand-indexed write blocks in `encode_observation` (316-578).
**Problem:** Every slot is a named `Final[int]` literal; the layout's correctness depends on ~60 of them staying mutually consistent across three *separate* size derivations (`_CONFIG_BLOCK_SIZE`, `_TIER_FLEET_BLOCK_SIZE`, `_SPEC_BLOCK_SIZE`) plus the `OBSERVATION_DIM = 73 + ... + 7 + ...` sum at line 127 whose `+7` and `+73` are themselves magic (the `+7` is hand-counted from the comment at 122-126). Insert a slot anywhere and you must hand-renumber every constant after it (this is exactly what v12 already did — "the specialization block slid one slot down 178->179..243->244" per the docstring). The single guard is the import-time `len(V1_ACTION_ORDER)==NUM_ACTIONS` assert in action_space.py; there is **no** assert that the highest `_S_*` index + its block size equals `OBSERVATION_DIM`, so an off-by-one in the constant wall produces a silently mis-aligned feature, not a crash.
**Code-judo remedy:** Replace the constant wall with a declarative feature registry: a tuple of `FeatureBlock(name, width)` entries whose offsets are computed by a running `accumulate`, and `OBSERVATION_DIM = sum(width)`. `encode_observation` writes via `obs[block.slice] = ...` instead of magic literals. Add one module-level assert that the registry's total equals the persisted `OBSERVATION_DIM`. This deletes ~60 `_S_*` constants and all three ad-hoc block-size derivations, makes inserts O(1) (append a block), and converts every future "slid one slot down" migration into a no-op. High effort but this is the single biggest fragility in the bucket.

### H3. Three duplicated play-set membership constants define overlapping-but-different play taxonomies across eligibility.py / mask.py / selector.py
**Location:** `eligibility.py:123` and `mask.py:68` (`_CANDIDATE_REQUIRED_PLAY_TYPES`, byte-for-byte duplicated — with an in-code comment at eligibility.py:120 admitting the duplication "to keep the import-direction contract"); `eligibility.py:691` (`live_confirm_plays` set, inline inside a staticmethod); `mask.py:91` (`_CIRCUIT_BREAKER_ELIGIBLE_PLAYS` derived from one copy).
**Problem:** The candidate-required set is the canonical example: the same 8-element frozenset is maintained in two files because the import-direction rule forbids mask.py-importing-eligibility-importing-mask. A play added to one and not the other produces an asymmetry between what the mask gates and what the authority confirms — a latent confirm/re-pick livelock source. `live_confirm_plays` (eligibility.py:691) is a *third* near-identical set (candidate-required minus GROOM_BACKLOG) built fresh on every `confirm()` call (hot path, per re-pick).
**Code-judo remedy:** Move `_CANDIDATE_REQUIRED_PLAY_TYPES` (and the `live_confirm_plays` variant) to `agentshore.play_rules` (already imported by both files — `needs_review`, `TERMINAL_SHUTDOWN_EVIDENCE_WINDOW_PLAYS` live there) and import from the one location. Hoist `live_confirm_plays` to a module-level frozenset so it isn't rebuilt per confirm. Removes one full duplicated frozenset + the apologetic comment, and collapses three taxonomies toward one.

### H4. `compute_reward` triplicates 19 reward terms: constant -> breakdown field -> raw sum -> log kwargs
**Location:** `reward.py:166-192` (19 `RewardBreakdown` fields), `reward.py:367-387` (the 19-line `raw = a + b + ... ` sum), `reward.py:399-424` (the 19-kwarg `_logger.debug` call), plus 6 scattered module-level bonus constants (43-89).
**Problem:** Adding or removing a reward term requires four coordinated edits (field, sum line, log kwarg, and usually a constant). The sum at 367 is a flat 19-term addition that must stay in lockstep with the dataclass — drop a term from the sum and it's silently excluded from `raw_total` while still shown in the breakdown. The log call re-types every field name a third time. This is the classic "scattered additive reward terms that should be a declarative list" anti-pattern.
**Code-judo remedy:** Make `RewardBreakdown` the single source: compute `raw = sum(getattr(bd, f) for f in _SUMMED_TERMS)` where `_SUMMED_TERMS` is the tuple of additive field names (excludes `concurrent_agent_utilization`/`_multiplier`/`raw_total`/`clipped_total`), and emit the debug log via `**dataclasses.asdict(bd)`. Deletes the 19-line manual sum and the 24-line manual log kwargs (~40 lines) and makes term changes a one-line edit. Consider a per-term `RewardTerm(name, fn)` table to also collapse the 19 `if ...: bd.x = ...` shaping blocks (228-365), but the sum/log fix alone is the high-conviction win.

### H5. `selector.py` is 1193 lines with `select()` carrying the entire re-pick state machine inline (~240 lines, 4 distinct continue-paths)
**Location:** `selector.py:285-526` (`select`), with config-head sampling (385-423), reverse-failsafe param prep (430-478), and live-confirm (491-503) all inlined into one loop body.
**Problem:** `select()` interleaves five concerns in one method: observation/mask build, the bounded re-pick loop, config-head sampling, reverse-failsafe annotation, and live-confirm. The four `continue` paths (claim-lost, failsafe-precondition-retry, confirm-reject, plus the implicit success return) each mutate `remaining_mask`, sometimes release a claim, sometimes bump `_last_select_repick_count` — the invariants are only documented in prose comments (e.g. 354-357, 446-448). Plus the file mixes in file-locking primitives (146-192), checkpoint pruning/archival (88-143), and global-canonical delta-merge I/O (953-1009) that have nothing to do with selection.
**Code-judo remedy:** (a) Extract the file-lock helpers (`_exclusive_file_lock`/`_lock_file`/`_unlock_file`/`_prepare_windows_lock_byte`, 146-192) and the checkpoint lifecycle (`_prune_local_checkpoints`/`_archive_old_canonicals`/`cleanup_stale_canonical_weights`/`_write_global_canonical_blocking`, 88-143 + 953-1009) into a `checkpoint_store.py` module — ~250 lines leave selector.py and gain independent testability. (b) Extract the config-head sampling block (385-423) into `_sample_config(...)` returning `(config_override, cfg_action, cfg_log_prob, cfg_mask)`. This shrinks `select()` by ~40 lines and isolates the dual-head logic. Net: selector.py drops under ~900 lines and the orchestration loop becomes readable.

## Medium

### M1. PR-pressure saturation constant (10.0) lives independently in three places and the obs uses a 4th literal
**Location:** `observation.py:144` (`_SAT_OPEN_PRS_COUNT = 10.0`), `reward.py:88` (`_PR_PRESSURE_MAX_OPEN_PRS = 10.0`), `reward.py:163` (`max_open_prs: float = 10.0` dataclass default). The obs at observation.py:539 and 419 both key off `_SAT_OPEN_PRS_COUNT` but the reward keys off its own copy.
**Problem:** Comments at reward.py:86 and observation.py:55-57 explicitly note "must mirror" — i.e. a known coupling maintained by hand. If the obs feature and the reward bonus ever disagree on "full", PPO sees a normalized pressure ratio that doesn't match the reward gradient it's being trained against.
**Code-judo remedy:** Define `SAT_OPEN_PRS_COUNT` once (config or a shared `rl/constants.py`, which already exists with one lonely constant) and import in both files. Removes two literals and the two "must mirror" comments.

### M2. `confirm()` rebuilds the entire candidate plan + agent-eligibility mask on every re-pick attempt — O(attempts x plan-build) on the hot path
**Location:** `eligibility.py:585-674` (`confirm`), called once per re-pick iteration in `selector.py:492`.
**Problem:** Each confirm does a fresh `build_candidate_plan(live_state)` (628), a fresh `compute_agent_eligibility_mask` (634), and a full `_validity_reason_for` re-derivation (643) — for a single play type. In a run of clean re-picks (up to `NUM_ACTIONS`=22 attempts), the plan is rebuilt up to 22 times per tick even when the live graph didn't change. The live-graph reload is the only thing that legitimately needs redoing.
**Code-judo remedy:** Cache the live plan + agent mask on the authority keyed by graph identity, or have the selector loop reload-and-rebuild once and pass the live plan into a lighter `confirm(play_type, params, live_plan)`. Confirm then only does the target-presence check (`_live_target_reason`, already a pure lookup). Removes the per-attempt plan rebuild.

### M3. `compute_config_mask` uses raw string status comparisons (`a.status.value == "error"`) instead of the `AgentStatus` enum used everywhere else
**Location:** `eligibility.py:916-928` (`"terminated"`, `"error"`, `"idle"` string literals), and `observation.py` throughout (e.g. 362-363, 374, 465-467 `a.status.value == "idle"`).
**Problem:** The same module's `compute_agent_eligibility_mask` (eligibility.py:830) uses the typed `AgentStatus.IDLE`; `compute_config_mask` 30 lines later uses bare strings. A typo in a status string is a silent always-false comparison (no enum-membership check). `last_error_class` magic strings (`"auth"`, `"invalid_model"`, `"rate_limit"`) are similarly stringly-typed across eligibility.py:807/922/925/928 and mask.py:586.
**Code-judo remedy:** Replace `.status.value == "..."` with `.status == AgentStatus.X` (the enum is already imported). Promote the `last_error_class` literals to an `ErrorClass` StrEnum or module constants. Mechanical, removes a whole class of silent-false bugs.

### M4. `MaskReason.__eq__` returning `NotImplemented` + str-coercion methods (`lower`/`upper`/`__contains__`) is a string-impersonation shim that invites silent type confusion
**Location:** `mask_reason.py:78-104`.
**Problem:** `MaskReason` deliberately quacks like `str` (compares equal to bare strings, supports `in`, `.lower()`, `.upper()`) so that "existing log emission sites keep working without change" (docstring 21). This is a `Any`-in-disguise: `mask_reason == "No IDLE agents"` type-checks and silently does substring-free equality, and `selector.py:194-216` (`_only_capacity_waiting`) does exactly this brittle substring matching (`if marker in reason for marker in capacity_markers`) against the `.text` — re-introducing the free-text matching the typed `classification` field was created to eliminate.
**Code-judo remedy:** The `classification` enum already exists; route `_only_capacity_waiting` and any other consumer through `reason.classification` / `reason.source` instead of substring tests, then delete the `lower`/`upper`/`__contains__`/str-eq shim from MaskReason. Removes the string-impersonation surface and the parallel `capacity_markers` literal list (selector.py:198-205).

### M5. `_loop_escalation` magic ladder duplicated between observation and reward semantics
**Location:** `observation.py:289-296` (`_loop_escalation`: 7->3, 5->2, 3->1) vs `reward.py:355-365` (loop penalty arms at fail_streak>=3, any_streak>=6).
**Problem:** Two independent hard-coded streak ladders (obs feature vs reward penalty) that both encode "when is a streak bad". They're meant to teach the policy the same thing from two sides but are tuned by separate literals with no shared definition; drift between them de-correlates the loop feature from the loop reward.
**Code-judo remedy:** Define the escalation thresholds once (the loop-detection ladder is referenced in MEMORY as issue #9 / the loop-detector) and derive both. Low urgency but flagged as a coupling.

### M6. `compute_terminal_no_work_config_mask` and `compute_config_mask` re-derive `effective_model_tier_config` / capability lookups with overlapping filter chains
**Location:** `mask.py:108-127` vs `eligibility.py:885-962`.
**Problem:** Terminal-QA config masking (mask.py) post-filters the base config mask for `tier=="large"` + `can_test` + not-excluded, partially re-implementing the per-config eligibility loop. The "large + can_test" QA-agent predicate is a magic policy embedded in the mask module.
**Code-judo remedy:** Express the QA-spawn predicate as a parameter to `compute_config_mask` (e.g. `require_capability="can_test", require_tier="large"`) rather than a second post-filtering function. Folds two functions into one parameterized one.

## Low

### L1. `metrics.py` imports moved inside function bodies for no stated reason
**Location:** `metrics.py:205` (`from agentshore.state import AgentStatus`), `metrics.py:308` (`from agentshore.state import AgentStatus as _AgentStatus`), `metrics.py:314` (`import time as _time`), `metrics.py:416` (`from datetime import ...`).
**Problem:** `AgentStatus` is imported twice inside two different functions (once aliased), `time` and `datetime` lazily inside loops. `agentshore.state` is already a top-level import in this module (line 14). These local imports add no laziness benefit (state is already loaded) and obscure dependencies.
**Code-judo remedy:** Hoist to module top. Trivial.

### L2. `recent` reassigned to two different windows in `_build_context`
**Location:** `metrics.py:263` (`recent = history[-_ROLLING_WINDOW:]`, window=10) then `metrics.py:292` (`recent = history[-10:]`, churn window — same 10 but a separate literal).
**Problem:** The name `recent` is rebound mid-function to a second window computed with a bare `10` literal that happens to equal `_ROLLING_WINDOW`. A future change to one and not the other silently desyncs rolling-stats and churn windows. `recent_created = 0` (293) with comment "same conservative heuristic" is a permanently-zero term added to a ratio — dead arithmetic.
**Code-judo remedy:** Use distinct names; use `_ROLLING_WINDOW` for both or name the churn window. Drop `recent_created` (always 0) from the churn formula at 296.

### L3. `_only_capacity_waiting` hard-codes `actionable_ignores = {"Reserved action slot"}` and a 6-entry substring marker list
**Location:** `selector.py:194-216`.
**Problem:** Decides log severity (warning vs debug) by substring-matching reason text against magic phrases ("No IDLE", "allowed tier", "instantiate_agent cooldown", ...). These strings must track the exact `MaskReason.text` literals emitted in eligibility.py; any reword silently flips the severity heuristic. (Subset of M4's string-matching root cause.)
**Code-judo remedy:** Switch to `reason.classification == TRANSIENT` and `reason.source == ELIGIBILITY/CONFIG`; delete the marker list.

### L4. `replay.py` reaches into `store._conn` (private attribute) for a raw SQL query
**Location:** `replay.py:103` (`async with self._store._conn.execute(...)`).
**Problem:** `ReplayLoader.iter_compatible_sessions` bypasses the DataStore API and runs raw SQL against the store's private connection — a layering violation that couples replay to the schema and breaks if DataStore changes its connection handling.
**Code-judo remedy:** Add a `DataStore.distinct_experience_session_ids(action_space_version)` method and call it. Removes the private-attribute reach.

### L5. `training.py` has four redundant `is None` re-checks after a combined guard
**Location:** `training.py:141-151`.
**Problem:** `config_active_any` already asserts `batch.config_active is not None`, then lines 142-151 re-raise `RuntimeError` for `config_actions`/`config_old_log_probs`/`config_masks`/`config_active` being None — the last (150-151, re-checking `config_active`) is provably dead since `config_active_any` (136-140) already dereferenced it. These are defensive asserts the type system should carry.
**Code-judo remedy:** Type `Batch`'s config fields so that when `config_active is not None` the rest are too (a small `ConfigBatch` sub-dataclass that's either all-present or absent), or collapse to a single guard. Removes 4 raise-blocks.

### L6. `INDEX_TO_PLAY` is redundant with `V1_ACTION_ORDER` tuple indexing
**Location:** `action_space.py:30`.
**Problem:** `INDEX_TO_PLAY = {i: pt for i, pt in enumerate(V1_ACTION_ORDER)}` is just `V1_ACTION_ORDER[i]`. A dict keyed by dense ints 0..21 duplicating a tuple. Minor.
**Code-judo remedy:** Use `V1_ACTION_ORDER[action]` at the one selector call site (selector.py:375) and delete the map. (`PLAY_TO_INDEX` is genuinely useful; this inverse is not.)
