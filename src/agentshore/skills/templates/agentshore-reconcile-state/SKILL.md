# agentshore-reconcile-state

Trunk-scoped local diagnosis from `$AGENTSHORE_PROJECT_PATH`. `$ARGUMENTS` is unused (the failure_streak ≥ 3 eligibility gate is the trigger). Read AgentShore's logs + DB + live process state, classify the wedge into a known pathology, then take **strictly local** remediation — no `git push`, no PR/issue mutations, no outbound GitHub calls. A pathology outside the known set is reported in the result block, not filed anywhere.

**Pre-flight.** Don't `git checkout`; read state without mutating yet.
1. Read `$AGENTSHORE_PROJECT_PATH/.agentshore/context.json` (session_id, repo, owner, learnings).
2. Read `.agentshore/contexts/<session_id>/play-<play_id>.json`. The orchestrator pre-writes `recent_wedge_signals` here — prefer it over re-deriving via SQL. Expected keys (best-effort): `stagnation_ticks`, `same_type_failure_streak`, `last_failed_play_type`, `last_failed_error`, `recent_timeouts[]`, `dirty_trunk_paths[]`, `orphan_worktree_paths[]`, `active_agents_in_flight[]` (each entry: `agent_id`, `agent_type`, `current_play_id`, `current_play_type`, `current_play_started_at`). **The `active_agents_in_flight` list is the authoritative cross-check for zombie classification — read it before classifying any process as a zombie.** The same JSON also carries the **worktree-protection lists** any destructive worktree op must honour: `active_worktree_paths[]` (absolute paths with a live work claim), `young_worktree_paths[]` (absolute paths created less than `worktree_min_age_hours` ago), and `worktree_min_age_hours` (default `3`). Default each to `[]` / `3` if absent.
3. If `.agentshore/session_start_dirty.json` exists, prefer it — it's the authoritative pre-session baseline captured at bootstrap, and survives DB corruption. Use its `modified_paths[].mtime_utc` and `summary` (`pre_session`, `mtime_cluster_span_seconds`).
4. Scan structured logs **across all sessions**, newest-first (`ls -t .agentshore/logs/agentshore-*.log | head -10`). Killed-prior-play evidence often lives in an earlier session's file. `tail -n 500` each and look for `cli_dispatch_start`, `agent_dispatch_timed_out`, `play_execution_timeout`, `play_completed:success=false`, `db_recovered_via_sqlite_recover`. Most recent matching evidence wins; stop once you've attributed every dirty file's mtime range.
5. Capture ground truth: `git status --porcelain`, `git worktree list --porcelain`, `ps -ax -o pid,ppid,etime,state,command | grep -E '(claude|uv run|pytest)' | head -50`.

**Diagnose.** Classify into one or more known pathologies. Only act on a classification with unambiguous evidence; if ambiguous, leave it alone and file a follow-up in Step 5.

- **`dirty_trunk_from_killed_mutator`** — `git status` shows `M/A/D/R` entries AND a trunk-scoped play (`CLEANUP`, `MERGE_PR`, `DESIGN_AUDIT`, `GROOM_BACKLOG`, `RUN_QA`, `WRITE_IMPLEMENTATION_PLAN`, `REFINE_TASK_BREAKDOWN`, `SEED_PROJECT`, `CALIBRATE_ALIGNMENT`) failed within a window matching the dirty-file mtimes. Primary signal is **mtime clustering** — capture mtimes (`stat -f "%Sm %N" -t "%Y-%m-%dT%H:%M:%S"` on macOS, `stat -c "%y %n"` on Linux), compute span and overlap with prior killed-play windows. Decision:

  | Cluster span | Matches prior killed-play window? | Decision |
  |---|---|---|
  | < 120s, many files | yes | classify (high confidence) |
  | < 120s, many files | no log (logs rotated) | classify if pre-session AND auto-fixer signature (medium) |
  | > 1h, sparse | any | DO NOT classify — likely user WIP |
  | < 120s, few files (≤ 3) | any | DO NOT classify alone — file follow-up |

  Secondary content signals (corroborate only): auto-fixer reformat patterns / lockfile churn (cleanup); new test files / deleted scaffolding (implementation plays).

  **Untracked artifacts are NOT this pathology.** `dirty_trunk_from_killed_mutator` covers **tracked** modifications (`M/A/D/R`) only. Untracked (`??`) debris splits by depth: files at the **repo root** are handled deterministically by the orchestrator (per-play reclaim + a session-start sweep that quarantines them under `.agentshore/reclaimed/<play_id>/`) — never touch or `git clean` a root `??` entry here; a root `owned_by_active_play: false` entry is orphaned debris the orchestrator's sweep will reclaim, so record it under `unrecognized_pathologies` for visibility and move on. Untracked debris in a **subdirectory** (path contains `/`) is unreachable by that root-only sweep and is handled by the separate `untracked_trunk_debris` pathology below. In all cases, any `recent_wedge_signals.dirty_trunk_paths` entry with `owned_by_active_play: true` is **in-flight work of a still-running trunk-scoped play, not a wedge** (#162): exclude it from every pathology, never act on it, and do not let it drive `success: false`.

  **Tracked in-flight edits (#224).** The orchestrator now also sets `owned_by_active_play` on **tracked** (`M/A/D/R`) `dirty_trunk_paths` entries whose mtime falls at/after a live trunk-scoped play's start. Treat a tracked entry with `owned_by_active_play: true` exactly like the untracked case above — it is in-flight work of a still-running play, not a wedge: exclude it from every pathology, never `git checkout -- ` it, and never let it drive `success: false`.

- **`misplaced_managed_plan_artifact`** (#224) — a tracked file shows as `M` but cannot be attributed to a killed prior play (the `write_implementation_plan` play is contractually read-only and must post its plan only as a GitHub issue comment; an errant one that wrote its output into the tree leaves exactly this state). You MAY restore such a file **without** a killed-play window, but **only** when its diff proves it is AgentShore's own managed output: run `git diff -- <path>` and confirm the added lines contain the literal header `AGENTSHORE_IMPLEMENTATION_PLAN`. The signature is the sole authorisation — if it is absent the dirty content is NOT a managed artifact and may be user work, so leave it untouched.

- **`conflicted_merge_in_progress`** — `$AGENTSHORE_PROJECT_PATH/.git/MERGE_HEAD` exists, or `git status --porcelain` shows unmerged entries (`UU`/`AA`/`DD`/`AU`/`UA`/`UD`/`DU`). A killed/errant `merge_pr` left the main checkout mid-merge with unresolved conflicts — this is the state that latches the trunk-dispatch pause. High confidence whenever `MERGE_HEAD` is present; no mtime analysis needed.

- **`untracked_trunk_debris`** — `git status --porcelain --untracked-files=all` shows an untracked (`??`) entry **below the repo root** (path contains `/`) that is not an AgentShore/beads sidecar. These are machine-generated artifacts a trunk-scoped play left in the shared checkout — a build/dependency lockfile, resolved manifest, compiled output, or codegen written into a subdirectory. At any depth they wedge trunk git operations: a later `git merge`/fast-forward that touches the same path aborts with `"untracked working tree files would be overwritten by merge… Aborting"`, permanently blocking `merge_pr`. The orchestrator's own untracked reclaim only sweeps **depth-1 root** scratch, so subdirectory debris is structurally unreachable by it — that gap is this pathology.
  - **Ownership check first (mandatory, do not skip).** Verify the file is not a running play's live output before considering it debris: `stat` the candidate's mtime and compare against every `active_agents_in_flight[]` entry whose `current_play_type` is trunk-scoped (`CLEANUP`, `MERGE_PR`, `DESIGN_AUDIT`, `GROOM_BACKLOG`, `RUN_QA`, `WRITE_IMPLEMENTATION_PLAN`, `REFINE_TASK_BREAKDOWN`, `SEED_PROJECT`, `CALIBRATE_ALIGNMENT`). If the file's mtime ≥ any such play's `current_play_started_at`, it is that play's **in-flight work — never classify or touch it**. (The `recent_wedge_signals.dirty_trunk_paths[].owned_by_active_play` flag also reflects this once the orchestrator computes ownership at depth, but do the `stat` check yourself regardless — it is the safety gate.)
  - **Attribution** (same discipline as `dirty_trunk_from_killed_mutator`): classify only when the mtime falls inside a **completed/killed** trunk-scoped play's window (from the Pre-flight log/DB scan) OR the path is listed in the `session_start_dirty.json` pre-session baseline. An untracked file that is > 1h old and isolated, or that you cannot attribute to any trunk-play window and is absent from the baseline, is possible user WIP: **do not classify** — record under `unrecognized_pathologies`.

- **`orphan_worktrees`** — `git worktree list --porcelain` shows paths under the project's worktree root with no active session row in `sqlite3 .agentshore/agentshore.db "SELECT worktree_path FROM worktrees WHERE session_id='<current>' AND status='active'"`.

- **`divergent_worktree_registration`** — git's recorded worktree path is **gone on disk and marked prunable**, but the actual working directory still exists under a *different* name whose `.git` file points back at that prunable admin dir (`gitdir: .../.git/worktrees/<name>`). This is a **moved/renamed working dir**, NOT an orphan: the registration path and the live directory diverged (#214 — git registers `.../pickup-17` as prunable-missing while `.../agentshore-17/.git` still references `.git/worktrees/pickup-17`). Do **not** treat it as an orphan — a blind `git worktree prune` here deregisters the admin dir the live directory depends on and breaks it. Evidence to capture: the prunable `git worktree list --porcelain` entry (path + `prunable` line), the dependent dir's `.git` file contents (`cat <dependent>/.git`), and whether the dependent dir is **in-flight** — cross-check `active_agents_in_flight[]` and `sqlite3 .agentshore/agentshore.db "SELECT play_id, completed_at FROM plays WHERE play_id = <play backing that worktree>"`. A dependent dir is in-flight when an `active_agents_in_flight` entry's worktree resolves to it OR its backing play has `completed_at IS NULL AND started_at IS NOT NULL`.

- **`zombie_subprocess`** — `ps` state `Z` AND ppid=1 (reparented) OR ppid matches a recent `cli_dispatch_start` whose play later emitted `agent_dispatch_timed_out`. Only classify if you can name the specific `play_id` and timeout event. **Critical additional constraint**: before classifying any PID as a zombie, cross-check that the associated `play_id` is NOT the current active in-flight play for that agent. An agent process is only a zombie if (a) it satisfies the `ps`/log criteria above AND (b) the orchestrator's current session does not have that play listed as actively in-flight right now. If the orchestrator DB shows `plays.completed_at IS NULL AND plays.started_at IS NOT NULL` for that `play_id` (meaning the play is still running), the process is NOT a zombie — it is a live agent backing an active play and must not be killed. Query: `sqlite3 .agentshore/agentshore.db "SELECT play_id, play_type, started_at, completed_at FROM plays WHERE play_id = <play_id>"`. If `completed_at` is NULL, the play is still active; skip this PID entirely and record it under `unrecognized_pathologies` for operator review instead.

- **`stale_worktree_lockfile`** — `lsof` shows a lockfile (`.git/index.lock`, `~/.cache/uv/.lock`, `.beads/lock`, etc.) held past the holding play's expected timeout, cross-referenced to `cli_dispatch_start` for the PID.

**Remediate.** Only act on classifications from above.
- **Conflicted merge in progress:** `git merge --abort` (the only merge operation this skill may run; it unwinds the in-progress merge and restores the pre-merge worktree — the conflicting work lives on the PR branch, not trunk). Verify `.git/MERGE_HEAD` is gone and `git status --porcelain` has no unmerged (`UU`/etc.) entries → `remediation.merge_aborted: true`. Do this BEFORE any per-path dirty-trunk restore, since unmerged paths can't be `git checkout -- `'d while the merge is live. Failure → `remediation.merge_aborted: false`, `success: false`.
- **Dirty trunk:** `git checkout -- <path>` per attributed path. Never `--force`/`--theirs`. Verify `git status --porcelain` clears the path → `remediation.trunk_paths_restored`. Failures → `trunk_paths_failed`, continue.
- **Misplaced managed plan artifact:** for a tracked path classified as `misplaced_managed_plan_artifact` above (its `git diff` added the `AGENTSHORE_IMPLEMENTATION_PLAN` header), `git checkout -- <path>` to revert it to HEAD even with no killed-play window. Record it under `remediation.trunk_paths_restored` and note `attribution: managed_plan_signature` in the diagnosis evidence. Skip (never restore) any tracked path whose diff lacks that header.
- **Untracked trunk debris:** dispose by **quarantine-move — never delete, never `git clean`/`rm`** (mirrors the orchestrator's own reclaim: files are *moved*, so content stays recoverable). Per classified path `<p>`: `mkdir -p ".agentshore/reclaimed/reconcile-<play_id>/$(dirname "<p>")"` then `mv "<p>" ".agentshore/reclaimed/reconcile-<play_id>/<p>"` (preserve the relative path so it can be restored). Verify `git status --porcelain` no longer lists `<p>` → record under `remediation.untracked_debris_quarantined`. A move that fails or a path that reappears → `remediation.untracked_debris_failed`, continue. Only ever quarantine paths **containing `/`** (below root); a root-level untracked file stays the orchestrator's job — never touch it here.
- **Worktree-removal safety guard (applies to EVERY `git worktree remove --force` below — orphan AND divergent-stale).** Before removing any worktree path, in this exact order: (1) **skip if `path` is in `young_worktree_paths`** — it was created less than `worktree_min_age_hours` ago and may belong to an agent dispatched *after* this skill's diagnosis snapshot (#218); (2) if `path` is not in `young_worktree_paths` but you cannot independently confirm from filesystem metadata that it is older than `worktree_min_age_hours`, **skip it** (`age_unknown`); (3) **skip if `path` is in `active_worktree_paths`** (a live work claim); (4) **re-read the live claim set immediately before the destructive command** — run `sqlite3 .agentshore/agentshore.db "SELECT worktree_path FROM worktrees WHERE session_id='<current>' AND status='active'"` again at remediation time (not just at diagnosis) and skip the path if it now appears. A path skipped by any of these is recorded under `worktrees_skipped` with the reason; it is `partial`, never `success: false`. This guard exists because diagnosis and removal are minutes apart and a worktree can be allocated to a freshly dispatched agent in that gap.
- **Orphan worktrees:** these are **registered** in git metadata, so a DB-only mark-stale leaves the registration behind and Verify can never pass. Remove the registration for real, preserving genuinely-uncommitted work — and only after the worktree-removal safety guard above clears the path. Per orphan path: check dirtiness with `git -C <path> status --porcelain` (best-effort).
  - **Clean** (empty porcelain, or git can't introspect the de-registered dir): `git worktree remove --force <path>` then `git worktree prune`. Verify the path is gone from `git worktree list --porcelain`, then mark the matching active DB row stale with `failure_reason='reconcile_state: orphan removed'` → `remediation.worktrees_removed`.
  - **Dirty** (porcelain shows uncommitted changes): do **not** destroy unsaved work. Leave the worktree in place, mark the DB row stale (`failure_reason='reconcile_state: orphan preserved (uncommitted changes)'`), and record the path under `remediation.worktrees_preserved_dirty`. A preserved-dirty orphan makes that category `partial`, not `success: false`.
- **Divergent worktree registration:** branch strictly on the in-flight check from diagnosis. **`git worktree prune` is forbidden in this pathology while any dependent dir is in-flight** — it would deregister the admin dir that dir depends on.
  - **Dependent dir is in-flight:** run `git worktree repair <dependent-dir>` (the only repair invocation this skill makes — it re-points git's registry at the moved working directory, non-destructively). Verify the prunable entry is gone and `git worktree list --porcelain` now shows the dependent dir as a normal (non-prunable) worktree → `remediation.worktrees_repaired`. Do **not** prune or remove afterward. Repair failure → leave untouched, record under `unrecognized_pathologies`, category `partial`.
  - **Dependent dir is stale** (no `active_agents_in_flight` match AND backing play `completed_at IS NOT NULL`, or no backing dir remains): the registration is genuinely dead — but **apply the worktree-removal safety guard above to `<dependent-dir>` first** (young / age-unknown / live-claim → skip as `partial`). Only if it clears: `git worktree remove --force <dependent-dir>` (skip if the dir is already gone) then `git worktree prune` to clear the prunable admin dir. Apply the orphan dirty-check too (`git -C <dependent-dir> status --porcelain`); if it shows uncommitted changes, preserve it like a dirty orphan instead (record under `worktrees_preserved_dirty`, `partial`). Verify both the prunable entry and the dependent dir are gone from `git worktree list --porcelain`, mark the matching DB row stale (`failure_reason='reconcile_state: divergent registration pruned'`) → `remediation.worktrees_removed`.
- **Zombies:** `kill -9 <pid>` only for provably AgentShore-owned processes where the DB confirms the backing play has already completed (`completed_at IS NOT NULL`) or timed out. A process whose `play_id` still has `completed_at IS NULL` in the DB is an active agent — do not kill it. Verify gone with `ps -p`. Record PID + 80-char command excerpt + the log line proving ownership + the `completed_at` value → `remediation.processes_killed`.
- **Stale lockfile:** confirm holder is AgentShore-spawned, kill (zombie rules), wait ≤ 5s for auto-clear. Only `rm` the lockfile if it's in `.agentshore/`, `.cache/uv/`, or a named AgentShore-managed path → `remediation.lockfiles_cleared`.

**Verify.** Re-run ground-truth checks; each remediated category must come back clean (or be reported `partial`). `git status --porcelain --untracked-files=all` (after filtering AgentShore sidecars) shows none of the paths you restored or quarantined — every `untracked_trunk_debris` path you quarantined is gone from the working tree (moved under `.agentshore/reclaimed/`); every `git worktree list` entry beyond main matches an active DB row, is a preserved-dirty orphan you recorded above (those leave the worktree category `partial`), or is a repaired worktree that now shows as a normal non-prunable entry; no prunable entry remains for a divergent registration you remediated; no zombie children of dispatch parents; no AgentShore-owned process holding a lock past its timeout. Record each check command + exit_code + summary in `verification_evidence`. Failure here → `success: false` with explanation in `error`.

**Unrecognized pattern.** If diagnosis saw a pathology outside the known set above, **do not file anything** — record it in the result block under `unrecognized_pathologies` (a one-line summary plus the supporting log/git/ps/lsof excerpts and `session_id`) so an operator can review it. No GitHub calls.

**Forbidden mutations:**
- Never create/edit/restore/delete `.github/workflows/**`, `.github/actions/**`, `.gitlab-ci.yml`, `.circleci/**`, `azure-pipelines.yml`, `Jenkinsfile`, `bitbucket-pipelines.yml`, or tests asserting their existence. Any such remediation → `success: false`, `error: "ci-change requested but forbidden by skill policy"`, files untouched.
- Never `git push`, and never `gh pr/issue create/close/edit/comment/merge` — this skill makes no GitHub mutations at all.
- Never `git stash` (entries leak across branches/sessions).
- Never `git worktree add` — the skill never creates worktrees. `git worktree remove --force` + `git worktree prune` are permitted **only** for orphan-worktree remediation (a registered worktree with no active session row) and for the **stale** branch of a divergent registration, and only after the dirty-check above. `git worktree repair <dir>` is permitted **only** for the **in-flight** branch of a divergent worktree registration (re-pointing git at a moved working dir); never run `prune` against an in-flight divergent registration. Otherwise `git worktree list` is read-only.
- Never `git reset --hard`, `git clean -f`/`git clean`, `rm` of a working-tree file, or `git checkout` with branch switching. Working-tree restore is `git checkout -- <path>` on **specific paths attributed to a killed prior play**, or on a tracked path whose diff carries the `AGENTSHORE_IMPLEMENTATION_PLAN` managed-artifact signature (see `misplaced_managed_plan_artifact` in Diagnose). A tracked path that is neither attributed nor signature-matched is never restored. Untracked debris is disposed **only** by quarantine-move (`mv` into `.agentshore/reclaimed/`) per the `untracked_trunk_debris` remediation — never `git clean`/`rm`, which destroy unrecoverably — and only for attributed subtree (`/`-containing) paths.
- Never `kill -9` a process you cannot prove (via the log) is a defunct child of a timed-out dispatch.
- Never `gh repo fork`, never `git remote add` a non-origin remote, and never open a cross-fork PR (a `gh pr create` whose `--head` points at a fork). This skill performs no push operations at all — the no-fork clause is listed here for completeness and to prevent any future workaround attempts.

`success: false` is reserved for a remediation this skill **performed** that then failed verification — a `git checkout -- ` that didn't clear its path, a quarantine-move (`untracked_trunk_debris`) whose path still shows dirty afterward, a `merge --abort` that left `MERGE_HEAD`, a worktree op that didn't take. It is **not** the outcome for dirty state the skill safely **declined** to touch. If the only unremediated state is dirt you cannot attribute to a known pathology (ambiguous ownership) or that is flagged `owned_by_active_play: true`, record it under `unrecognized_pathologies` and return `success: true` (use `partial` where a category was only partly handled): reconcile correctly took no action, and the dirt is either in-flight work or transient teardown debris (e.g. tracked edits from agents being killed during a fleet contraction) that a later quiescent reconcile — or the orchestrator — will clear. This is the #224 fix: a safe no-op must not burn the play as a failure. A run that diagnosed nothing actionable is likewise `success: true` with empty `remediation` and `pathologies`. **A run whose only dirty state is untracked root artifacts (`??` at the repo root), an unattributable subtree untracked file left untouched as possible user WIP, or tracked/untracked edits flagged `owned_by_active_play: true`, is exactly this no-op `success: true` case, never `success: false`.** Always emit the block.
