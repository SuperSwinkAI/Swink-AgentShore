AgentShore RL Architecture

An RL-based orchestrator that coordinates multiple LLM coding agents using Proximal Policy Optimization to select high-level development actions.

PPO · 22 actions · 246-dim state · policy version 5

Core Loop

A Agent

AgentShore manages multiple LLM coding agents as async subprocesses (CLI) or HTTP clients (API). Each agent has a lifecycle, circuit breaker, and context budget.

Agent typesclaude-code, codex, gemini, gpt, other
Lifecycleidle → busy → terminated
Circuit breaker3 failures / 300s, 60s cooldown
Max context200k (CC), 192k (Codex), 1M (Gemini)
Anti-confirmationreviewer ≠ author (hard constraint)

E Environment

The environment is a GitHub repository with open issues, pull requests, a running codebase, and a beads epic/story/task graph. AgentShore observes repository state, dispatches agent actions, and measures alignment against beads task closure.

  • GitHub issues synced via gh CLI
  • Beads graph synced from GitHub issue state
  • Alignment measured by global and per-epic closure ratios
  • Budget tracked in real dollar cost
  • Session persisted in SQLite (20 tables, WAL mode)

S State — Observation Vector

A 246-dimensional float32 vector (OBSERVATION_VERSION = 13) encoding the full session state, normalized to [0, 1] via saturation clipping. Fed to the policy network at each decision step. Slots 0–71 are the base feature block; slots 72–167 are the per-config block; slots 168–178 are PR-author and execution-pressure features; slots 179–244 are the per-tier / per-play specialization block; slot 245 is a version marker.

  • 8 legacy zero-filled slots plus 4 beads epic/global closure slots
  • 5 issue metrics (open, closed, created, net velocity, scope completion)
  • 16 fleet metrics (3 tiers × idle/busy/success/context/total + active count)
  • 4 budget features (remaining, spent, avg cost, sufficiency flag)
  • 16 history features (last-5 play types, last-5 success flags, rolling stats, drift)
  • 3 time features (session duration, since calibration, since seed)
  • 3 PR state, 4 health, 2 handoff, 3 trajectory, 3 learnings, 1 churn
  • 96 per-config slots (32 agent configs × idle/busy/success-rate)
  • 4 PR-author features (open + awaiting-review by Claude / Codex)
  • 5 execution-pressure features (rolling velocity, busy fraction, unreviewed-PR ratio, mergeable-PR ratio, in-flight pickups)
  • 1 version marker = OBSERVATION_VERSION / 10

A Action Space — 20 Plays

Discrete action space of 20 high-level development operations. Actions are masked at each step based on preconditions (e.g., can't merge if no approved PR exists). The policy outputs logits over all 20; masked actions receive −∞ before softmax. instantiate_agent is evaluated per type/tier: busy agents can justify another spawn, while an idle same-config agent blocks that config.

π Policy Network

Shared-trunk ActorCritic MLP with separate heads for action logits and state value. Action masking sets forbidden logits to −∞ before softmax.

Parameters~51K (shared trunk + 3 heads; +~4K when the config head is active)
Headsactor (22 plays), value V(s), config (N agent configs)
ActivationReLU (hidden), linear (heads)
InferenceCPU-only, <50ms

V Value Function

The value head estimates V(s) — the expected discounted return from the current state. Used in Generalized Advantage Estimation (GAE) to compute advantage targets for the policy update.
