Proposal - desktop packaging and lifecycle control

AgentShore Desktop Layer Cake

The desktop application should be a lifecycle control plane for AgentShore, not only a packaged browser dashboard. Tauri owns native app concerns, AgentShore owns orchestration semantics, and the dashboard remains a state surface on top of AgentShore's existing IPC.

Companion documents: desktop onboarding and startup flows (mockups) and DESIGN.md (resolved design decisions). When this page and DESIGN.md disagree, DESIGN.md wins.

Primary user promise Choose a repo, choose the target branch, prepare AgentShore, start a session, and monitor it in one app.
Hard boundary The desktop UI talks to a versioned machine protocol, not human CLI output.
Runtime posture Prefer TCP loopback IPC in packaged builds so Windows and macOS share one path.

The Stack

Layer 6

Platform bundle and installer

Tauri emits the app bundle, DMG, MSI, or NSIS installer. The release artifact contains the native shell, the frozen AgentShore sidecar, and the dashboard assets already copied into the Python package static directory.

macOS .app DMG MSI NSIS signing
Layer 5

Tauri WebView experience

The visible application: project picker, target branch setup, setup checklist, session start screen, dashboard tab, logs, settings, and health panels. It should expose explicit controls for project preparation and session start behavior.

start screen dashboard logs settings
Layer 4

Tauri Rust supervisor

The native host owns process lifecycle, bundled binary paths, child shutdown, platform menus, window state, file dialogs, and app updates. It is the only layer allowed to spawn or kill the AgentShore sidecar.

spawn kill events file dialog updater
Layer 3

AgentShore desktop IPC sidecar

A long-lived Python sidecar exposes a versioned machine API for project inspection, preparation, start, stop, status, logs, and health checks. This layer translates desktop commands into AgentShore domain operations without scraping human CLI text.

JSON-RPC 2.0 stdio versioned structured errors
Layer 2

AgentShore core session

The existing orchestrator loop, play executor, RL selector, agent manager, SQLite store, and StateProvider pipeline. In desktop mode it should run in agent IPC mode with TCP loopback so the dashboard and control surfaces share a portable endpoint.

core loop plays agents SQLite StateProvider
Layer 1

Project and external tool surface

The user's selected repository, Git, beads, GitHub auth, configured coding agents, API keys, and local filesystem. The desktop app should detect and explain missing prerequisites instead of bundling every external tool.

Git beads GitHub Codex Claude API keys

Guiding Rule

The CLI remains human friendly. The desktop path gets a stable, typed, versioned machine contract. That keeps Tauri from depending on prompt text, terminal formatting, or incidental Click output.

Desktop Flow State Model

Choose project Primary action: choose a repository folder.
Readiness Inspect target safety, Git, GitHub, existing state, and local capabilities.
Target branch Select and validate the branch AgentShore treats as trunk for QA, cleanup, merges, and new work.
GitHub identities Choose token-backed GitHub logins and trusted external PR-author logins.
Agent runners Enable runner types and assign GitHub logins from the login pool.
Runner detail Select one available model for each small, medium, and large tier.
Start session Choose runtime settings; app applies setup choices before launch.
Starting Show structured progress: config preparation, IPC, session, dashboard bridge.
Dashboard Open the live dashboard after the first AgentShore state snapshot arrives.
End Session Report Drain, stop, or completion lands on ESR with report, logs, archive, and restart.

Desktop Protocol Boundaries

Boundary Protocol Responsibility Examples
WebView to Rust Tauri commands and events Fast UI actions, file dialogs, window events, process status. select_project, start_session, stop_session
Rust to sidecar JSON-RPC 2.0 over stdio Machine API for AgentShore lifecycle operations and structured errors. project.inspect, project.branches, session.start, session.stop
Core to dashboard Existing AgentShore IPC plus WebSocket bridge Real-time state snapshots, play events, and validated dashboard commands. state_update, play_event, feedback_requested
Sidecar to external tools Existing AgentShore subprocess and API adapters Interact with Git, beads, GitHub, coding agents, and API providers. bd, gh, agent CLIs, OpenAI-compatible APIs

Machine API Shape

  • JSON-RPC 2.0 over stdio between Tauri Rust and the Python sidecar.
  • Long-running calls emit LSP-style $/progress notifications keyed by a token.
  • Cancellation via LSP-style $/cancelRequest notification.
  • Handshake exchanges protocol_version, agentshore_version, build_id, and a capabilities list; build_id mismatch is fatal.

Packaged Runtime

  • Freeze Python with PyInstaller --onedir for debuggability; revisit --onefile later.
  • Bundle bd (beads CLI) alongside Python; detect Claude / Codex / Gemini.
  • Drop textual, beaupy, and playwright from the desktop bundle; pin PyTorch to CPU-only wheels.
  • TCP loopback for packaged IPC, even on macOS, to keep parity with Windows.
  • Tauri 2 native shell ships dashboard assets as bundled resources, copied from the Python static dir at build time.
  • Tauri 2 updater + GitHub Releases manifest; macOS and Windows code-signed.

Non-Goals

  • Do not rewrite AgentShore core in Rust for the desktop shell.
  • Do not make the dashboard responsible for process supervision.
  • Do not parse Click output as an application protocol.
  • Do not expose project preparation as a standalone screen when it can follow setup choices.

Resolved Decisions

The questions originally raised here have been resolved through a structured design interview. Full rationale and dependencies live in DESIGN.md. Summary:

Question Resolution
Does the desktop sidecar run the orchestrator in-process or spawn the existing CLI? Hybrid: implement RPC methods case-by-case using internal AgentShore APIs; the JSON-RPC contract is designed independently. Single Python process; orchestrator runs as an asyncio task in the sidecar.
Can policy inference avoid shipping full training dependencies? Trim non-desktop deps (textual, beaupy, playwright); pin PyTorch to CPU-only wheels. Inference-only profile deferred until measurement justifies it.
How should multiple projects and sessions be represented? One active session at a time. One sidecar per app lifetime; project.select(path) switches context. Recents owned by the sidecar at platformdirs.user_data_dir('agentshore')/recents.json.
Which prerequisites are bundled versus detected? Bundle AgentShore Python sidecar and bd (beads CLI). Detect coding-agent CLIs (Claude, Codex, Gemini) with guided remediation on the Readiness screen.

Acceptance Bar For MVP

  • The app can select a repo and produce a structured inspect result without starting AgentShore.
  • The app can select and validate the target branch before setup is applied.
  • The app can collect GitHub logins and assign them to agent runners before setup is applied.
  • The app can configure small, medium, and large model choices for each runner before a live session starts.
  • The app can apply project preparation behind the scenes after setup choices are known.
  • The app can start AgentShore in agent IPC mode and open the dashboard after readiness.
  • The app can stop or drain the session and land on the End Session Report.
  • All lifecycle controls use structured protocol messages, not terminal output parsing.
  • Crash and restart paths preserve logs and do not orphan supervised child processes.
