v0.15.0 phases — what shipped

Phase 1 · Typed mask reasons shipped

src/agentshore/rl/mask_reason.py defines MaskReason (frozen dataclass: text, classification, source), MaskClassification (TRANSIENT / INDEFINITE_WAIT / HARD), and MaskSource (9 gate identifiers). String-compatible so existing log paths and substring tests keep working. Orchestrator._mask_reason_is_transient and _is_indefinite_wait now read .classification directly — a one-line attribute check rather than substring matching.

Phase 2 · Typed override entries shipped

src/agentshore/plays/override.py defines OverrideEntry with five producer kinds: BOOTSTRAP (never drops on mask), USER_REQUEST (no retry budget, surfaces drops to the requester), EXECUTOR_REQUEUE (bounded transient retries), RETRY (auto-failure retry, budget already enforced), and MASK_REQUEUE (preserves the original producer's classification through any number of mask cycles). All seven producer sites in core/ wrap their enqueues in typed entries; _handle_masked_override dispatches on entry.kind instead of substring-matching the reason text.

Phase 3 · Reverse failsafe as overlay shipped

compute_reverse_failsafe_mask takes an optional base_mask keyword and returns lifted | base_mask — a structural superset of the base mask. Reverse failsafe can ADD opportunities to the selection space, never REMOVE them. Both production call sites (compute_action_mask for auto-failsafe, PPOSelector.select for manual) pass base_mask. The Phase 6 test suite pins the contract: test_reverse_failsafe_overlay_is_structural_superset_of_base_mask.

Phase 4 · One state per tick shipped

State is built once at the top of each tick and threaded through selection → dispatch → executor as a single object. _dispatch_play takes state: AgentShoreState as a parameter and no longer calls _build_state. The executor's precondition recheck is intentionally retained as a safety net AND as the signal source for _recent_executor_skip — when the same state object reaches the executor, the only way preconditions can disagree with the mask is a contract bug elsewhere, which the skip-rate metric now surfaces.

Phase 5 · Action space v12 + skip metric shipped

Two new active actions, two new reserved slots; NUM_ACTIONS=22, ACTION_SPACE_VERSION=12.

  • IDLE_TICK — productive wait when ≥1 agent is BUSY. Zero cost; skipped_outcome("waiting"). Lets PPO learn to wait one tick instead of forcing reverse failsafe.
  • RECOVER — eligible only when state.recent_executor_skip is set. Zero cost; no-op execute. Gives PPO an explicit recovery action after an executor-time masked skip.
  • FUTURE_7, FUTURE_8 — reserved headroom (always hard-masked).

New observation feature: executor_skip_rate_recent_50 at slot 177 (Orchestrator tracks a 50-outcome deque; rate exposed to MetricsEngine via the executor_skip_rate_provider callback). OBSERVATION_DIM grew 238 → 245 (action-space growth + skip-rate slot); OBSERVATION_VERSION bumped 9 → 10. policy.py:load_checkpoint rejects mismatched action_space_version / policy_version / observation_version with a CheckpointVersionMismatch; _archive_old_canonicals renames legacy checkpoints to policy_legacy_v{N}.pt so the next run starts cleanly.

Phase 6 · Mask composition pipeline + tests shipped

compute_action_mask is a documented 10-stage pipeline of pure _stage_* functions: 8 zero-only mutations (preconditions, agent-eligibility, forced-zeros, candidate-required, instantiate-config, end-session, take-break, reserved-slots) plus 2 short-circuits (drain-mode, terminal-no-work). The reverse-failsafe overlay (Phase 3) is the final step. Per-stage unit tests live in tests/test_rl_mask.py — test_stage_preconditions_seeds_from_registry, test_stage_forced_zeros_is_zero_only, test_stage_reserved_slots_zeros_future_slots, test_stage_drain_mode_short_circuits_to_end_agent, and the composite invariant test test_pipeline_order_invariant_short_circuit_after_zero_only.

Hotspots — status after v0.15.0

HOTSPOT 1 · Cross-tick state divergence addressed

The structural cause is removed. _build_state is called once per tick; the same state object is threaded through selection, dispatch, and the executor. The dispatch-time rebuild that previously produced play_skipped_masked_at_executor events is gone.

Executor preconditions backstop: the recheck at executor.py:240 stays — it's the only defence if a future bug introduces disagreement between compute_action_mask and play.preconditions(state) on the same state object. The recheck now serves a second purpose too: it emits the _recent_executor_skip signal that drives the RECOVER action and the new executor_skip_rate_recent_50 observation feature.

HOTSPOT 2 · Reverse-failsafe candidate-required gate fixed v0.14.4 · strengthened v0.15.0

v0.14.4 added the _CANDIDATE_REQUIRED_PLAY_TYPES zero-out so reverse failsafe can't conjure a target out of nothing. v0.15.0 added the overlay contract: compute_reverse_failsafe_mask(base_mask=mask) returns lifted | base_mask, a guaranteed structural superset of the base mask. Both production callers pass base_mask; the contract is pinned by a Phase 6 test.

HOTSPOT 3 · Override drop on cooldown fixed v0.14.4 · typed v0.15.0

The substring-matching predicate is gone. _handle_masked_override reads entry.kind first — BOOTSTRAP entries never drop, regardless of what the mask says — then entry.enqueue_classification. The bootstrap medium-of-different-type spawn cannot be dropped by predicate drift because there is no predicate.

HOTSPOT 4 · Precondition fail at executor time addressed

The structural cause is removed by Phase 4 (one state per tick). The remaining occurrences are now observable and learnable:

  • The orchestrator tracks a rolling 50-outcome window and exposes the rate via _executor_skip_rate_recent_50().
  • MetricsEngine reads it via executor_skip_rate_provider.
  • ObservationContext.executor_skip_rate_recent_50 is written to observation slot 177 — PPO can learn the gating signal directly.
  • state.recent_executor_skip drives RECOVER's preconditions — PPO can pick an explicit recovery action instead of relying on reverse failsafe.

Open questions / things to watch

  • IDLE_TICK starvation risk. Reward shaping should ensure IDLE_TICK is only attractive when truly no other play is viable. The current cold-start weight is 0.0114 (low), but PPO can learn arbitrarily high preference — watch the play history in early sessions.
  • RECOVER cycling. If executor-time divergence persists across multiple ticks (unlikely now that state is threaded), RECOVER would re-fire on each tick. The signal clears on the next non-skipped completion, so this should be self-limiting; verify in a live session.
  • Skip-rate observation usefulness. Slot 177 is informational for PPO. If divergence is effectively zero after Phase 4 (as expected), the slot stays at 0 and provides no learning signal. That's the desired outcome — but it also means the slot is "dead weight" most of the time. Consider whether to reclaim the slot in v0.16 if divergence stays at zero for a month.
  • Phase 1 string-MaskReason fallback. _mask_reason_is_transient and _is_indefinite_wait still accept MaskReason | str. The string branch is a fallback for any unmigrated gates. Plan to delete it once a session log shows zero bare strings hitting either method.
  • Bootstrap diversity post-cooldown. Phase 5 routes bootstrap entries through OverrideKind.BOOTSTRAP which never drops on mask. Verify the medium-of-different-type spawn fires after seed + cooldown in the next live session.
  • Checkpoint reset blast radius. OBSERVATION_VERSION 9 → 10 and ACTION_SPACE_VERSION 11 → 12 both reject prior checkpoints. _archive_old_canonicals renames them to policy_legacy_v{N}.pt; the next session starts with fresh cold-start weights. Current policy has minimal training so the cost is bounded — but document this in release notes.
