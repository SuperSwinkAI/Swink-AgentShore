AgentShore / Dispatch Anatomy

What's actually in the envelope when AgentShore triggers an agent.

Spawning a coding agent is one line of shell. Spawning one that knows what to do is the whole job.

You can shell out to claude. Or codex. Or gemini. The process starts in whatever directory your terminal happens to be in. It signs commits with whatever git config it inherits. It reads whatever CLAUDE.md is closest on disk. It picks an issue to work on by — well, you didn't tell it which one, so it picks. Sometimes the same one twice. Sometimes one that doesn't exist anymore. Sometimes it edits .github/workflows/ because nobody told it not to.

I've been running this loop overnight, and while I was at the day job, for months. Every piece on this page is a thing I learned by waking up to it being broken.

Most of the work isn't generating code. It's not letting the agent inherit ambient state it didn't earn.

01 / Before pickupAn issue gets groomed through four or five plays before any agent gets to touch code.

By the time the envelope is sealed, the issue inside it has already been through a pipeline that operates on the beads graph — not on source files. None of these earlier plays ship a PR. Skip them, and the implementer ends up burning tokens guessing at scope on an ambient-from-GitHub issue with no acceptance criteria, no parent epic, and no dependency edges.

Fig. 1 · The pre-pickup pipeline. PPO picks which stage to run next based on play preconditions — no hard sequencer enforces order, but the gates make the order de-facto emergent.

02 / The envelopeEvery dispatch is six pieces, assembled, then handed over.

When PPO picks a play and the parameter resolver picks the agent and the issue, this is the shape of what gets pushed across the subprocess boundary. Nothing here is typed by hand at dispatch time. Each piece has a source the rest of the system is already maintaining.

Fig. 2 · The six pieces of context AgentShore assembles per dispatch. None of them come from the shell environment the user happened to be in.

03 / Inside the context fileThe structured side-channel the agent reads before it does anything.

The biggest piece in the envelope — the rendered SKILL.md — is natural language. But the agent also reads a structured JSON snapshot of the world AgentShore thinks it's stepping into. The path is injected via --append-system-prompt "Context file: …", and the rendered prompt instructs the agent to open it first. The file is written through a temp-file + os.replace(), so a half-written payload can never be observed by a racing reader.

04 / Where the pieces come fromFour systems, already maintaining themselves, fold into one envelope.

The trick isn't building any one of these pieces. It's keeping their sources separated so each one can be authoritative for its own slice. Beads owns the task graph. GitHub owns the human conversation. agentshore.yaml owns identity. AgentShore's SQLite owns the RL state and the per-session memory.

Fig. 4 · The assembler doesn't author context. It indexes systems that already track it — and reassembles the envelope every dispatch.

05 / What the agent doesn't get to inheritThe shortest path to a broken loop is letting the subprocess pick its own state.

Fig. 5 · The expensive surprises overnight all live on the left. The boring discipline on the right is what makes the loop self-driving.

06 / After the envelope is openedThe dispatch isn't done when the subprocess exits.

Every play ends with a post-hoc issue-inflation check via validate_scope(), a reward computed against the beads alignment delta, and a row appended to dispatch_replay so the dispatch can be reconstructed later. The same machinery that built the envelope reads it back — which is the only reason the next dispatch knows anything about this one.

Persistence you don't need beats lost context. So everything gets written down, even when it feels like overkill.

None of this is glamorous. It's mostly about not letting one agent's ambient state become the next agent's bug. Naming the working directory. Naming the identity. Naming the issue. Naming the scope. Writing it all down so the next loop iteration can read it back.

I just want the agents to work the backlog like an employee — show up at the right repo, sign as themselves, work the right issue, stay in their lane, and leave a note for whoever picks up after them. Still solving my own problems the way I like them solved. TBD if others like it.
